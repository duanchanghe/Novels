# 📚 项目开发计划 — EPUB 转 AI 有声小说

## 1. 项目概述
- **项目名称**：AI 有声书工坊（AI Audiobook Workshop）
- **项目愿景**：让每一本电子书都能被"听见"，用 AI 生成媲美真人主播的超高品质有声小说
- **项目目标**：用户将 EPUB 电子书放入指定文件夹后，系统自动监听、自动解析、自动合成、自动发布，实现从"电子书"到"有声书"的全自动化流水线
- **核心哲学**：
  - 🎯 **零操作自动化**：放书进去，拿有声书出来，中间无需任何人干预
  - 🎯 **章节独立生成**：每章独立生成独立的音频文件，支持单章试听/下载/替换
  - 🎯 **超高品质标准**：角色区分 + 情感表现 + 专业级音频后处理，媲美真人录制
  - 🎯 **自动分发**：生成完成后自动发布到有声书平台
- **目标用户**：
  - 🎧 喜欢"听书"的阅读爱好者（通勤、睡前、做家务时）
  - 👨‍🦯 视障人士，需要无障碍阅读工具
  - ✍️ 网络小说作者，想为自己的作品制作有声版
  - 📖 出版社/有声书平台，批量将电子书转为有声内容
- **核心指标**：
  - 语音合成自然度 MOS 评分 ≥ 4.0（行业优秀线）
  - 单本 10 万字小说全自动生成时长 ≤ 30 分钟
  - 角色区分准确率 ≥ 90%
  - 零人工干预完成率 ≥ 95%
  - 音频采样率 ≥ 44.1kHz，比特率 ≥ 192kbps（CD 级音质）
- **技术栈**：
  - **后端**：Python FastAPI + Celery（异步任务队列）
  - **前端**：Next.js (TypeScript)
  - **数据库**：PostgreSQL
  - **LLM（角色识别/文本处理）**：DeepSeek API
  - **TTS（语音合成）**：MiniMax TTS API
  - **文件存储**：MinIO（兼容 S3 协议）
  - **消息队列**：Redis
  - **容器化**：Docker + docker-compose

## 4. 阶段0：需求分析与方案设计

### 4.1 需求收集与分析
- [ ] 明确产品业务背景与目标
- [ ] 调研竞品（Audible、喜马拉雅、微信读书等）功能与体验
- [ ] DeepSeek LLM 能力评估：
  - 中文对话/旁白识别准确率
  - 角色关系推理与说话人归因
  - 上下文窗口能否处理长章节文本
  - API 调用成本与并发限制
- [ ] MiniMax TTS 能力评估：
  - 支持的音色数量与风格（如男声/女声/童声）
  - 中文发音自然度与多音字处理
  - 是否支持情感/语气参数调节
  - API 调用成本、并发限制与延迟
  - 单次请求最大字符数与音频时长
  - 最高输出音频质量参数（采样率、比特率、声道数）
- [ ] MinIO 存储方案评估：
  - 本地部署 vs 容器化部署
  - 存储容量规划（预估每本书 500MB~2GB）
  - 访问策略（临时链接、CDN 集成）
- [ ] **🔔 文件夹自动监听方案调研**：
  - **方案一：文件系统事件监听（推荐）**
    - Python `watchdog` 库：监听 `books/incoming/` 目录的 `on_created` 事件
    - 优点：实时响应，无需轮询，资源消耗低
    - 缺点：需保证监听进程持续运行
  - **方案二：定时扫描（兜底方案）**
    - Celery Beat 定时任务：每 30 秒扫描一次 `books/incoming/` 目录
    - 优点：实现简单，与 Celery 生态集成
    - 缺点：有延迟，不适合高实时性场景
  - **方案三：WebHook 回调**
    - 外部文件同步工具（如 Syncthing）同步完成后回调 API
    - 适用场景：多设备同步
  - **最终方案**：**方案一（watchdog）+ 方案二（Celery Beat 兜底扫描）** 双重保障
- [ ] **📤 有声书平台自动发布方案调研**：
  - **目标平台分析**：
    - 喜马拉雅：开放平台 API（需申请开发者权限，支持音频上传、专辑管理）
    - 蜻蜓 FM：开放平台 API（类似喜马拉雅）
    - 荔枝 FM：开放平台 API
    - Apple Books / Audible：需 iTunes Connect 或 ACX 平台
    - 自有平台（自建站点）：直接通过 MinIO 预签名 URL 分发
  - **发布策略**：
    - **初级阶段**：自建平台 + 预签名 URL 直链分发
    - **中级阶段**：接入喜马拉雅/蜻蜓 FM 开放平台 API
    - **高级阶段**：多渠道同时发布（一键分发到 3-5 个平台）
  - **发布内容**：
    - 单章音频文件（MP3，44.1kHz，192kbps+）
    - 完整有声书 M4B（含章节标记）
    - 封面图、简介、章节元数据
- [ ] **🎯 超高品质标准定义**：
  - **音频技术指标**：
    - 采样率：44.1kHz（CD 标准）或 48kHz（广播级）
    - 比特率：192kbps（MP3 高质量）或 320kbps（MP3 极限质量）
    - 声道：立体声（必要时支持 5.1 环绕）
    - 信噪比（SNR）≥ 85dB
    - 总谐波失真（THD）≤ 0.1%
  - **听感指标**：
    - 无电子音/机械感
    - 角色音色区分度明显
    - 情感表达自然不夸张
    - 段落间停顿节奏合理
    - 多音字/人名/地名发音准确
    - 数字/日期/英文朗读自然
  - **文件规范**：
    - 命名：`{书名}_第{章节号}章_{章节标题}.mp3`
    - ID3 标签：包含书名、作者、章节名、封面图、专辑信息
    - M4B 格式：含章节书签，支持播放器自动跳转

  - 作为用户，我想预览任意章节的语音效果，以便确认质量
  - 作为用户，我想下载完整有声书（MP3 / M4B 格式），以便离线收听
  - **作为用户，我希望把 EPUB 放进指定文件夹就自动开始生成，无需任何操作**
  - **作为用户，我希望每章生成独立的音频文件，方便分章收听**

  - **作为用户，我希望生成完成后自动发布到我听书的平台**

### 6.5 模块D：超高品质音频后处理与 MinIO 存储
> **核心能力**：将 MiniMax 合成的原始音频片段，经过专业级音频处理流水线，输出 CD 品质的有声书文件
- [ ] **音频拼接引擎**（`pydub` + `ffmpeg` 协同）：

  - **交叉淡入淡出（Crossfade）**：相邻片段间 10-30ms 交叉淡入淡出，消除拼接痕迹
  - **智能停顿控制**：
    - 句号/问号/感叹号 → 0.5s 停顿
    - 段落分隔 → 0.8s-1.5s 停顿（根据情感强度自动调节）

- [ ] **音量均衡处理（专业级）**：
  - **LUFS 标准化**：将整体响度标准化到 -16dB LUFS（ITU-R BS.1770 标准）
  - **峰值限制（Limiter）**：防止爆音，峰值不超过 -1dB TP
  - **动态范围压缩（Compressor）**：轻压缩（Ratio 2:1, Threshold -20dB），平衡对话与旁白音量差
  - **EQ 均衡**：轻微提升 3kHz-5kHz 频段（增强语音清晰度），削减 200Hz 以下低频（减少喷麦/低频噪声）
- [ ] **降噪处理**：
  - 噪声门（Noise Gate）：消除静音段的底噪
  - 自适应降噪（轻度）：保留语音频段（300Hz-3.4kHz）
- [ ] **音频格式转换**：
  - **MP3 高质量**：320kbps CBR, 44.1kHz, Stereo（最大兼容性）
  - **MP3 标准**：192kbps CBR, 44.1kHz, Stereo（平衡质量与大小）
  - **M4B 有声书格式**：AAC LC 256kbps, 44.1kHz，含章节书签标记

    - 封面图嵌入

## 7. 阶段3：超高品质音质优化与多角色合成（核心攻坚）

> 这是本项目最具技术挑战、也最能体现产品差异化的环节。目标是让 AI 合成语音达到"听不出是 AI"的沉浸式体验。

### 7.1 MiniMax TTS 深度调优
- [ ] **全面测试 MiniMax 所有可用音色**：
  - 自然度 MOS 评分（5 人以上盲听团队，每音色 10 个样本）
  - 中文发音准确度（多音字、轻声、儿化音、变调、前后鼻音）
  - 不同语速下的音质稳定性（0.8x ~ 1.5x）
  - 长文本（>1000 字）合成一致性
  - 情感参数真实效果对比（中性 vs 高兴 vs 悲伤 vs 愤怒）
  - 建立**音色评测矩阵**：音色ID × (自然度/清晰度/情感表现力/稳定性)
- [ ] **DeepSeek 文本预标准化深度优化**：
  - 数字朗读规则：
    - 年份："1949年" → "一九四九年"（非"一九四九"年）
    - 电话号码：逐位朗读
    - 百分比："50%" → "百分之五十"
    - 小数："3.14" → "三点一四"
  - 日期/时间规则：
    - "3月5日下午2点30分" → 保持原样（但统一格式）
    - "2024-03-05" → "二零二四年三月五日"
  - 英文单词判断（DeepSeek 上下文分析）：
    - 专有名词（如 iOS, AI, WiFi）→ 保持英文发音
    - 普通英文词 → 判断是否为中文文本中的英文借词
  - 特殊符号处理：
    - 破折号（——）→ 转换为 0.3-0.5 秒停顿
    - 省略号（...）→ 转换为 0.5-0.8 秒停顿，且语速渐慢
    - 感叹号（！）→ 保持，但判断是否是真正感叹语气（如"你好！"vs"小心！"）

## 21. 编码规范要求

> 本项目要求**组件化、结构化、格式化**编码，便于审查排错和后续拓展修改。

### 21.1 通用规范
- [ ] **中文备注**：所有核心逻辑代码必需有完整详实的中文备注（docstring + 行内注释）
  - 文件头部：说明文件用途、作者、创建日期
  - 类定义：说明类职责、核心方法、使用示例
  - 函数/方法：说明参数、返回值、异常、注意事项
  - 复杂逻辑段：行内注释解释"为什么这么做"而非"做了什么"
- [ ] **编码格式**：
  - Python：UTF-8 编码，使用 4 空格缩进
  - TypeScript：UTF-8 编码，使用 2 空格缩进
  - 行尾统一 LF（Unix 风格）
- [ ] **命名规范**：
  - Python：`snake_case`（函数/变量），`PascalCase`（类），`UPPER_CASE`（常量）
  - TypeScript：`camelCase`（函数/变量），`PascalCase`（类/组件/类型），`UPPER_CASE`（常量）
  - 文件名：`snake_case.py`（Python），`kebab-case.ts`（TypeScript）
  - 自解释命名：禁止单字母命名（除循环计数器 i/j/k）

### 21.2 组件化要求
- [ ] **每个模块封装为独立组件**：
  - 一个 Python 文件只包含一个核心类（或一组紧密相关的函数）
  - 对外暴露统一接口（`__init__.py` 清晰导出）
  - 内部实现细节不对外暴露（使用 `_` 前缀私有方法）
  - 通过依赖注入解耦（不使用全局变量/单例）
- [ ] **目录结构组件化**：
  ```
  backend/services/
  ├── __init__.py          # 统一导出所有服务
  ├── svc_epub_parser.py   # EPUB 解析组件
  ├── svc_minimax_tts.py   # TTS 合成组件
  └── ...                  # 每个服务一个文件
  ```
- [ ] **前端组件化**：
  - 每个页面组件拆分为：容器组件（Container）+ 展示组件（Presentational）
  - 公共组件放在 `components/common/` 目录
  - 页面专属组件放在 `components/{page_name}/` 目录
  - 每个组件一个目录：`ComponentName/index.tsx` + `ComponentName/styles.module.css`

### 21.3 结构化要求
- [ ] **三层架构**：API（路由层）→ Service（业务层）→ Model（数据层）
  - API 层：只负责 HTTP 请求/响应处理，不包含业务逻辑
  - Service 层：纯业务逻辑，不依赖 HTTP 上下文
  - Model 层：数据模型 + 数据库操作，不包含业务逻辑
- [ ] **错误处理结构化**：
  ```python
  # 统一的异常层次结构
  class AppError(Exception):                    # 所有自定义异常基类
  class FileError(AppError):                    # 文件相关异常
  class EPUBParseError(FileError):              # EPUB 解析异常
  class TTSApiError(AppError):                  # TTS API 异常
  class DeepSeekAPIError(AppError):             # DeepSeek API 异常
  class PublishError(AppError):                 # 发布异常
  ```

- [ ] **配置结构化**：
  - 所有配置从环境变量读取（12-Factor App）
  - 配置文件分层：`config/base.py` → `config/dev.py` → `config/prod.py`
  - 敏感信息（API Key/密钥）不硬编码，通过 `.env` 文件注入

- [ ] **代码格式化工具**：
  - Python：Black（自动格式化）+ Ruff（Lint）+ mypy（类型检查）
